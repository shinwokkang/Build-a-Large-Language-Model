# 📖 [심층 해설] 2.5: 바이트 쌍 인코딩 (Byte Pair Encoding, BPE) 완벽 정복

지난 2.4절의 마지막 부분에서 아주 중요한 '충격적인 진실'을 예고했습니다. 과거에는 사전에 없는 모르는 단어가 나오면 `<|unk|>`(Unknown) 토큰으로 덮어버렸지만, 현대의 거대 언어 모델(GPT, Llama 등)은 이 방식을 절대 쓰지 않는다고 했습니다. 단어의 의미가 100% 날아가 버리기 때문이죠.

그렇다면 최신 AI들은 도대체 사전에 없는 신조어, 오타, 외계어를 어떻게 완벽하게 처리하는 걸까요? 그 마법 같은 비밀이 바로 이번 2.5절의 주제인 **BPE (Byte Pair Encoding)**, 즉 **서브워드(Subword) 토큰화 기법**입니다.

---

## 1. 🚀 혁신의 시작: 단어를 통째로 모르면, 쪼개서라도 알아내자!

과거의 토크나이저는 단어를 띄어쓰기 기준으로 통째로 저장했습니다. (`apple`, `banana` 등). 하지만 세상의 모든 단어, 모든 언어의 변형을 딕셔너리에 다 담을 수는 없습니다.

**BPE의 핵심 철학**: 
> "사전에 `someunknownPlace`라는 긴 단어가 통째로 없다고 포기하지 마라. 대신 내가 아는 더 작은 조각들인 `some`, `unknown`, `Place` 로 쪼개면 다 아는 단어들 아니냐!"

이처럼 BPE는 단어를 통째로 외우는 대신, 자주 쓰이는 글자들의 **'조합(Subword)'**을 기억합니다. 덕분에 아무리 낯선 단어가 들어와도 에러(`KeyError`)를 내지 않고, 자기가 아는 가장 큰 조각들로 레고 블록처럼 분해하여 완벽하게 인식해 냅니다.

---

## 2. 💻 실전 코드 분석: OpenAI의 `tiktoken` 라이브러리 도입

BPE 알고리즘을 파이썬으로 밑바닥부터 짜는 것은 꽤 복잡하고 계산량이 많습니다. 그래서 실무에서는 오픈소스 라이브러리를 사용합니다. 책에서는 OpenAI가 직접 만들어 배포한 초고속 토크나이저 라이브러리인 **`tiktoken`**을 사용합니다. (`tiktoken`은 속도가 매우 빠른 Rust 언어로 밑바탕이 짜여 있습니다.)

### 📝 코드 1: tiktoken 설치 및 버전 확인
```bash
pip install tiktoken
```
```python
from importlib.metadata import version
import tiktoken

print("tiktoken version:", version("tiktoken"))
# 출력 예시: tiktoken version: 0.7.0
```
* **해설**: 실무 환경에서 파이썬 패키지를 설치하고, 코드가 꼬이지 않도록 버전을 확인하는 표준적인 절차입니다.

### 📝 코드 2: GPT-2 토크나이저 불러오기
```python
# OpenAI의 GPT-2 모델이 실제로 사용했던 BPE 토크나이저를 그대로 불러옵니다.
tokenizer = tiktoken.get_encoding("gpt2")
```
* `get_encoding("gpt2")` 명령 한 줄이면, 우리가 앞서 2.3절에서 끙끙대며 만들었던 단어 사전 구축과 특수 문자 정규식 처리 등을 할 필요 없이 수만 개짜리 완성된 GPT-2 사전을 메모리에 띄울 수 있습니다.

### 📝 코드 3: BPE 인코딩 실습 (문자를 숫자로)
```python
text = (
    "Hello, do you like tea? <|endoftext|> In the sunlit terraces"
     "of someunknownPlace."
)

# allowed_special 매개변수를 통해 특수 토큰인 <|endoftext|>를 문자열 그대로 인식하게 허용합니다.
integers = tokenizer.encode(text, allowed_special={"<|endoftext|>"})
print(integers)
```
* **결과 (숫자 배열)**: `[15496, 11, 466, 345, 588, 8887, 30, 220, 50256, 554, 262, 4252, 18250, 8812, 2114, 286, 617, 34680, 27271, 13]`

### 📝 코드 4: BPE 디코딩 실습 (숫자를 다시 문자로)
```python
strings = tokenizer.decode(integers)
print(strings)
```
* **결과 (문자열)**: `Hello, do you like tea? <|endoftext|> In the sunlit terraces of someunknownPlace.`
* 인코딩했던 숫자를 그대로 디코딩했더니 글자 하나 틀리지 않고 100% 원본 텍스트로 복원되었습니다.

## 3. 🧠 코드 결과에서 발견하는 BPE의 2가지 놀라운 특징

방금 돌려본 코드 결과에서 우리는 두 가지 중요한 사실을 짚고 넘어가야 합니다.

1. **거대한 사전과 `<|endoftext|>`의 정체**: 
   - 배열 중간에 있는 `50256` 이라는 거대한 숫자가 바로 `<|endoftext|>` 토큰의 ID입니다. 
   - 즉, GPT-2의 단어 사전(Vocabulary)은 0번부터 50,256번까지 **총 50,257개**의 크기를 가지고 있다는 것을 유추할 수 있습니다. (우리가 2.3절에서 만든 1,130개짜리 꼬마 사전과는 차원이 다릅니다.)
2. **모르는 단어도 완벽하게 통과 (`someunknownPlace`)**:
   - `someunknownPlace`라는 단어는 세상 어떤 영어 사전에도 없는 억지로 이어붙인 신조어입니다. 
   - 우리가 2.3절에서 만든 토크나이저였다면 당장 `KeyError`를 냈겠지만, BPE는 에러 없이 숫자로 바꾸었고, 디코딩할 때도 완벽하게 되살려 냈습니다. 도대체 어떻게 한 걸까요?

## 4. 🖼️ BPE의 마법 해부하기 (Figure 2.11 해설)

그림 2.11과 Exercise 2.1은 BPE가 미등록 단어(OOV)를 어떻게 쪼개어 먹는지 직관적으로 보여줍니다.

* 타겟 단어: `"Akwirw_ier"` (완전한 외계어)
* 통째로 딕셔너리에 찾아보면 당연히 없습니다. 그래서 BPE는 이 단어를 자기가 아는 서브워드(조각)들로 찢기 시작합니다.
  - `Ak` (Token ID: 33901)
  - `w` (Token ID: 86)
  - `ir` (Token ID: 343)
  - `w` (Token ID: 86)
  - ` ` (공백) (Token ID: 220)
  - `ier` (Token ID: 959)

이처럼 BPE는 단어 전체를 모르면 두 세 글자로, 그것도 안 되면 알파벳(Character) 하나 단위로까지 철저하게 분해해서라도 기어코 사전에 있는 ID 값들로 치환해 냅니다. 덕분에 현대의 LLM은 `<|unk|>` 토큰을 단 하나도 쓰지 않고도 세상의 모든 문자를 다 받아낼 수 있게 되었습니다.

## 5. 🛠️ BPE 사전은 어떻게 만들어지는가? (작동 원리 요약)

그렇다면 BPE는 50,257개짜리 딕셔너리에 어떤 조각들을 넣을지 어떻게 결정한 걸까요? 사람이 일일이 잘라준 것이 아닙니다. 기계가 통계적으로 스스로 학습한 결과입니다.

1. **초기 세팅**: 세상의 모든 개별 알파벳(`a`, `b`, `c`...)과 기호들로만 기본 사전을 만듭니다. (절대 에러가 나지 않는 최소 단위)
2. **반복 병합 (Merging)**: 엄청나게 많은 텍스트를 읽으면서, 가장 자주 붙어 다니는 알파벳 쌍을 하나의 토큰으로 합칩니다. 
   - 예: `d` + `e` 가 엄청 자주 나오네? ➡️ `de`를 사전에 추가!
   - `de` + `f` 가 또 자주 나오네? ➡️ `def`를 사전에 추가!
3. **결과**: 이렇게 수만 번 반복하면, `the`, `apple`, `computer` 같이 흔한 단어는 통째로 사전에 남게 되고, 자주 안 쓰이는 외계어나 신조어는 알파벳 쪼가리로 남게 됩니다.

## 6. 🌐 [2027년 최신 실무 트렌드] BPE의 진화와 한국어 토큰화 문제

우리가 다룬 `gpt2`의 BPE 사전 크기는 50,257개였습니다. 하지만 기술이 발전하면서 현대의 거대 언어 모델들은 사전 크기를 대폭 키웠습니다.

* **[최신 트렌드 1] 거대해진 사전 크기 (Vocab Size)**:
  - 2027년 현재 쓰이는 GPT-4o, Llama 3, 4 모델 등은 사전 크기가 무려 **100,000 ~ 200,000개**를 훌쩍 넘습니다. 
  - 사전을 키우면 신조어도 통째로 인식할 확률이 높아지므로 문장이 훨씬 적은 개수의 토큰으로 쪼개집니다. ➡️ 텍스트를 더 빨리 읽고, 모델의 API 비용(토큰당 과금)도 저렴해지는 엄청난 실무적 이점이 있습니다.
* **[최신 트렌드 2] 다국어(Multilingual) 문제와 바이트(Byte) 레벨 처리**:
  - 기존 GPT-2 시절에는 영어를 기준으로 BPE가 학습되어, 한국어 "안녕하세요"를 넣으면 글자가 무참히 쪼개져 토큰이 수십 개로 낭비되는 현상(한글 패널티)이 심했습니다.
  - 최신 모델들은 훈련 데이터를 전 세계 언어로 확장하고, 알파벳 단위가 아니라 컴퓨터의 가장 기본 단위인 **바이트(Byte)** 레벨에서 BPE를 시작합니다. 이를 BBPE(Byte-Level BPE)라고 부르며, 어떤 언어나 이모티콘(😊)이 들어와도 절대 깨지거나 뻗지 않고 완벽하게 숫자로 통역해 냅니다.

### 💡 모범 답안 (정확한 이해를 위한 체크)

**답안 1**
BPE는 단어를 띄어쓰기 기준으로 통째로 인식해야 한다는 고정관념을 버렸습니다. 모르는 긴 단어(예: someunknownPlace)가 들어오면, 자기가 사전에 이미 가지고 있는 더 작은 단위의 조각들(Subword)이나 심지어 개별 알파벳(Character) 단위로까지 단어를 잘게 쪼갭니다. 이렇게 분해된 조각들은 항상 사전에 존재하기 때문에, 의미를 아예 잃어버리는 `<|unk|>` 치환 없이 원본 단어의 형태소와 철자 정보를 100% 보존한 채로 숫자로 변환(인코딩)할 수 있습니다.

**답안 2**
BPE 알고리즘의 유일한 판단 기준은 방대한 텍스트 훈련 데이터셋 내에서의 **"출현 빈도수(Frequency)"**입니다. 가장 자주 함께 연속해서 등장하는(frequently occur together) 두 개의 글자나 서브워드 쌍을 찾아서 하나로 병합(Merge)합니다. 이 과정을 수만 번 반복하면 'the'나 'ed' 같이 일상에서 가장 많이 쓰이는 조합은 하나의 완성된 토큰으로 사전에 등재되고, 잘 안 쓰이는 고유명사는 알파벳 파편으로 남게 됩니다.

**답안 3**
사전 크기를 키우면 더 많은 단어나 신조어, 심지어 한국어나 일본어 같은 다국어 단어들까지 사전에 통째로(하나의 토큰으로) 등재될 확률이 높아집니다. 똑같은 문장을 읽어도 예전에는 10개의 서브워드 토큰으로 쪼개졌다면, 최신 모델은 단 3~4개의 토큰만으로 문장을 통째로 인식할 수 있습니다. 토큰 개수가 줄어들면 모델이 글을 읽고 쓰는 연산 속도가 극적으로 빨라지며, API 사용료가 철저히 토큰 개수에 비례하여 부과되는 실무 환경에서 서비스 운영 비용을 획기적으로 절약할 수 있습니다.
