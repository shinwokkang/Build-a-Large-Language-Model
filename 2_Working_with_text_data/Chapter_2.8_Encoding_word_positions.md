# 📖 [심층 해설] 2.8: 단어의 위치 정보 인코딩 (Encoding word positions) 완벽 정복

지금까지 2장을 달려오면서 텍스트를 쪼개고(BPE), 정수 번호표(Token ID)를 붙이고, 이를 다차원 실수 벡터(Token Embeddings)로 변환하는 데 성공했습니다. 이제 당장이라도 LLM에 데이터를 밀어 넣고 싶겠지만, 트랜스포머(Transformer) 모델에는 아주 치명적인 설계상 결함이 하나 숨어 있습니다. 

바로 모델이 단어의 **'순서(Position)'를 전혀 모르는 순서 맹(Position-agnostic)**이라는 사실입니다. 2.8절에서는 이 약점을 극복하기 위해 단어 벡터에 위치표(나침반)를 달아주는 '위치 임베딩(Positional Embeddings)'의 개념과 코드를 완벽하게 파헤쳐 봅니다.

---

## 1. 🧠 단어 임베딩과 어텐션의 치명적인 약점 (Figure 2.17 해설)

우리가 2.7절에서 만든 `torch.nn.Embedding` (단어 임베딩 계층)은 본질적으로 거대한 '단어장(서랍장)'이라고 했습니다. 
그림 2.17이 이 단어장의 한계를 명확히 보여줍니다.

* **동일한 토큰 ID = 동일한 벡터**: 
  - `"사과"`라는 단어가 문장의 **맨 첫 번째(1번 위치)**에 나오든, **맨 마지막(100번째 위치)**에 나오든 모델은 똑같은 서랍(예: 5번 서랍)을 열어 똑같은 값을 꺼내옵니다.
* **어텐션(Self-attention)의 맹점**:
  - 다음 3장에서 배울 LLM의 뇌인 '어텐션 메커니즘'은 단어들을 한 줄로 세워놓고 순서대로 읽는 것이 아니라, 문장 안의 모든 단어들을 한 번에 통째로 붓고 섞어서(Bag-of-words 처럼) 연산합니다.
  - 따라서 모델은 **"개가 사람을 물었다"** 와 **"사람이 개를 물었다"** 를 수학적으로 완벽히 똑같은 문장으로 인식하는 끔찍한 오해를 저지르게 됩니다. 단어의 종류는 똑같기 때문이죠.

이 치명적인 문제를 해결하기 위해, 우리는 원래의 단어 벡터에 **"너는 몇 번째 자리에 있는 단어야"**라는 정보를 강제로 주입해야 합니다.

---

## 2. 🗺️ 위치 정보를 주입하는 두 가지 철학 (Absolute vs Relative)

위치 정보를 주입하는 방식에는 크게 두 가지 학파가 있습니다.

1. **절대 위치 임베딩 (Absolute Positional Embeddings)**:
   - "너는 무조건 1번 자리야", "너는 5번 자리야" 라고 각 고정된 자리(Index)마다 고유한 명찰(벡터)을 따로 만들어서 붙여주는 방식입니다.
   - OpenAI의 GPT-2, GPT-3 모델들이 전통적으로 채택해 온 방식입니다.
2. **상대 위치 임베딩 (Relative Positional Embeddings)**:
   - 절대적인 번호가 아니라, "나(A 단어)를 기준으로 너(B 단어)는 3칸 뒤에 있네" 처럼 단어들 사이의 상대적인 거리(Distance)에 집중하는 방식입니다.
   - 훈련 때 본 적 없는 엄청나게 긴 문장이 들어와도 유연하게 대처할 수 있다는 강력한 장점이 있습니다.

이번 실습에서는 GPT 모델의 근본을 따라 **'절대 위치 임베딩'**을 코드로 구현해 봅니다.

## 3. 💻 코드로 보는 절대 위치 임베딩 생성과 결합 (Figure 2.18 해설)

<img width="655" height="241" alt="Figure 2 18" src="https://github.com/user-attachments/assets/e831696b-f4f0-4b4d-8f46-4ca5b1b4c7cc" />


먼저 앞선 2.6절과 2.7절의 코드를 합쳐서, 단어 임베딩 텐서를 뽑아내는 과정까지 빠르게 복기해 봅시다.

### 📝 코드 1: 단어 임베딩 텐서 준비 (복습)
```python
import torch

vocab_size = 50257  # BPE 토크나이저의 사전 크기
output_dim = 256    # 임베딩 벡터 차원 (실제 GPT-3는 12288 차원이지만, 실습을 위해 256으로 축소)
max_length = 4      # 모델이 한 번에 읽을 수 있는 문맥 길이 (창문 크기)

# 단어 임베딩 계층 생성
token_embedding_layer = torch.nn.Embedding(vocab_size, output_dim)

# DataLoader에서 가져온 (배치 사이즈 8, 길이 4) 크기의 토큰 ID 텐서가 있다고 가정 (inputs)
# inputs.shape => torch.Size([8, 4])

# 토큰 ID를 256차원 벡터로 룩업(Lookup) 변환
token_embeddings = token_embedding_layer(inputs)
print(token_embeddings.shape) 
# 출력: torch.Size([8, 4, 256])
```
* **결과 상태**: 총 8개의 문장이 있고, 각 문장에는 4개의 단어가 있으며, 각 단어는 256개의 숫자로 표현되어 있습니다. 하지만 이 256개의 숫자 안에는 단어의 '뜻'만 들어있고 '순서'는 없는 상태입니다.

### 📝 코드 2: 위치 임베딩(Positional Embedding) 계층 생성
이제 '순서표'를 나눠주는 새로운 서랍장(Embedding Layer)을 하나 더 만듭니다.
```python
context_length = max_length  # 4

# 위치 임베딩 계층 생성 (크기: 4개의 자리 x 256차원)
pos_embedding_layer = torch.nn.Embedding(context_length, output_dim)

# torch.arange(4)는 [0, 1, 2, 3] 이라는 텐서를 만들어냅니다.
# 즉, 0번 자리, 1번 자리, 2번 자리, 3번 자리에 해당하는 위치 벡터 4개를 서랍에서 뽑아옵니다.
pos_embeddings = pos_embedding_layer(torch.arange(context_length))
print(pos_embeddings.shape)
# 출력: torch.Size([4, 256])
```
* **해설**: `pos_embeddings` 안에는 0번째 자리 특유의 패턴(256차원), 1번째 자리 특유의 패턴(256차원)... 이렇게 총 4개의 위치 좌표계 벡터가 생성되어 있습니다. 이 벡터들 역시 처음엔 랜덤값이지만 훈련을 거치며 최적의 위치 좌표계를 스스로 형성하게 됩니다.

### 📝 코드 3: 덧셈(+) 연산으로 단어 의미와 위치 정보 융합하기
가장 신기하면서도 중요한 순간입니다. 그림 2.18에서 보듯, 우리는 단어 벡터와 위치 벡터를 그냥 **더합니다(+).**
```python
# 단어의 뜻(8 x 4 x 256) + 단어의 위치(4 x 256)
input_embeddings = token_embeddings + pos_embeddings

print(input_embeddings.shape)
# 출력: torch.Size([8, 4, 256])
```

### 🍎 [아주 쉬운 비유와 예시] 단어와 위치의 덧셈, 도대체 어떻게 진행될까?

코드로만 보면 잘 와닿지 않을 수 있습니다. 256차원이나 되는 거대한 숫자는 머리 아프니까, 아주 간단한 **2차원 세계**로 축소해서 구체적인 숫자로 직접 계산해 봅시다.

#### 1단계: 단어 임베딩 (단어장)
우리 사전에 딱 두 단어만 있다고 가정해 봅시다.
* **"사과"** = `[10, 20]` (사과의 뜻을 나타내는 고유 벡터)
* **"바나나"** = `[30, 40]` (바나나의 뜻을 나타내는 고유 벡터)

#### 2단계: 위치 임베딩 (순서표)
문장에서의 자리를 나타내는 번호표도 2차원으로 만들어 봅니다.
* **0번 자리 (첫 번째)** = `[1, 1]`
* **1번 자리 (두 번째)** = `[2, 2]`

#### 3단계: 문장 입력과 덧셈 연산!
자, 이제 두 개의 서로 다른 문장(배치)이 입력되었다고 상상해 봅시다.
* 문장 A: **"사과 바나나"**
* 문장 B: **"바나나 사과"**

**문장 A ("사과 바나나")의 계산:**
1. 첫 번째 단어 **"사과"** (0번 자리):
   `[10, 20]` (사과) + `[1, 1]` (0번 자리) = **`[11, 21]`**
2. 두 번째 단어 **"바나나"** (1번 자리):
   `[30, 40]` (바나나) + `[2, 2]` (1번 자리) = **`[32, 42]`**
👉 결과: `[[11, 21], [32, 42]]`

**문장 B ("바나나 사과")의 계산:**
1. 첫 번째 단어 **"바나나"** (0번 자리):
   `[30, 40]` (바나나) + `[1, 1]` (0번 자리) = **`[31, 41]`**
2. 두 번째 단어 **"사과"** (1번 자리):
   `[10, 20]` (사과) + `[2, 2]` (1번 자리) = **`[12, 22]`**
👉 결과: `[[31, 41], [12, 22]]`

#### 💡 결과 분석: 왜 완벽한 해결책일까?
잘 보세요! 문장 A에 있는 "사과"는 **`[11, 21]`**이 되었지만, 문장 B에 있는 "사과"는 **`[12, 22]`**가 되었습니다. 
* 숫자의 큰 틀(10번대, 20번대)은 유지되므로 **"아, 이 단어는 사과구나"**라는 '의미'를 잃지 않습니다.
* 동시에 끝자리 숫자가 미세하게 달라졌으므로 모델은 **"아, 아까 사과는 0번 자리에 있었는데 지금 사과는 1번 자리에 있구나"**라는 '위치'를 완벽하게 구별해 냅니다.
이렇게 더하기(+) 연산 하나만으로 **'단어의 정체성'**과 **'문장 속 위치'**가 기가 막히게 하나로 융합되는 것입니다!

---

* **💡 파이토치 브로드캐스팅(Broadcasting)의 마법**: 
  - 방금 우리가 문장 A와 문장 B를 계산할 때, 위치 번호표 `[[1, 1], [2, 2]]`를 A에도 써먹고 B에도 똑같이 재사용해서 더했죠?
  - 파이토치도 마찬가지입니다. `token_embeddings`는 3차원(`8 x 4 x 256`)이고 `pos_embeddings`는 2차원(`4 x 256`)입니다. 크기가 안 맞지만, 파이토치는 첫 번째 배치의 4개 단어에 위치 벡터 4개를 순서대로 덧셈하고, **두 번째 배치, 세 번째 배치에도 이 위치 벡터 4개를 알아서 복사(재사용)해서 덧셈**해 줍니다. 이것이 바로 메모리를 획기적으로 아껴주는 브로드캐스팅 기능입니다.
* **💡 왜 더하기(+) 인가요? (이어 붙이지 않고?)**:
  - 벡터 두 개를 이어 붙여서 `[10, 20, 1, 1]` 처럼 4차원(512차원)으로 만들 수도 있습니다(Concatenation). 하지만 딥러닝 연구자들은 그냥 차원끼리 덧셈을 해도 방금 위에서 본 것처럼 고차원 공간에서는 원래 단어의 의미 정보와 위치 정보가 서로 간섭하지 않고 독립적으로 잘 보존된다는 사실을 증명해 냈습니다. 덧셈을 쓰면 차원이 커지지 않으므로(256 유지) 연산 메모리를 절반으로 아낄 수 있는 엄청난 이점이 있습니다.

## 4. 🚀 전체 데이터 전처리 파이프라인의 완성 (Figure 2.19 및 Chapter 요약)

그림 2.19는 길고 길었던 2장의 여정을 한 장의 그림으로 아름답게 요약해 줍니다. 인간의 텍스트가 어떻게 기계의 혈관(Transformer)으로 주입되는지 그 전체 파이프라인이 완성되었습니다.

1. **Input text**: "This is an example." (원본 텍스트)
2. **Tokenized text**: `["This", "is", "an", "example", "."]` (BPE 서브워드로 쪼개기)
3. **Token IDs**: `[40134, 2052, 133, 389, 12]` (단어 사전(Vocabulary)을 통해 정수로 변환)

5. **Positional embeddings (+)**: 각 자리 번호(0, 1, 2, 3...)에 해당하는 256차원 위치 벡터를 불러와 **덧셈(+)**

## 5. 🌐 [2027년 최신 실무 트렌드] RoPE (회전 위치 임베딩)의 시대

책에서는 GPT-3의 뼈대인 '절대 위치 임베딩(Absolute)'을 배웠습니다. 하지만 2027년 현재 실무 생태계는 완전히 뒤바뀌었습니다. 최신 Llama 3/4, Mistral, Gemma 등 모든 최고 성능 오픈소스 모델들은 절대 위치 임베딩을 버리고 **RoPE (Rotary Positional Embedding, 회전 위치 임베딩)** 이라는 기술을 표준으로 채택하고 있습니다.

* **RoPE의 개념 (왜 위대한가?)**:
  - 기존 덧셈 방식은 문장 길이가 훈련 때 설정한 `max_length`를 넘어가면 모델이 뻗어버립니다. (예: 4000번 자리 위치 벡터를 본 적이 없으므로 계산 불가)
  - RoPE는 위치 벡터를 덧셈(+)하는 것이 아니라, 단어 벡터를 복소수 평면에서 특정 **각도(Angle)만큼 회전(Rotation)**시키는 기하학적 방식을 씁니다. 
  - 1번 자리 단어는 10도 회전, 2번 자리는 20도 회전... 이렇게 회전시켜 두면, 단어와 단어 사이의 '상대적 거리(각도 차이)'를 모델이 수학적으로 우아하게 인지할 수 있습니다.
* **실무적 파급력**: 
  - RoPE 덕분에 모델 훈련 시에는 8,000 토큰까지만 학습시켰어도, 실전 배치 후 코드를 조금만 튜닝(NTK-Aware Interpolation 등)하면 모델을 재학습시키지 않고도 100만(1M) 토큰 길이의 책을 한 번에 읽어낼 수 있게 되었습니다. 2027년 무한 컨텍스트(Infinite Context) 시대의 1등 공신이 바로 이 RoPE 기술입니다.

파이토치 코드로 위치 임베딩을 결합할 때, `input_embeddings = token_embeddings + pos_embeddings` 처럼 두 텐서를 이어 붙이지(Concatenation) 않고 단순 덧셈(+)을 수행합니다. 덧셈을 수행하면 단어의 의미 정보와 위치 정보가 섞여서 망가질 것 같지만 실제로는 정상 작동하며 실무 표준으로 쓰입니다. 그 이유는 무엇인가요?

### ❓ 질문 3
GPT-3와 같은 초기 모델은 각 자리에 고유한 벡터를 할당하는 '절대 위치 임베딩(Absolute Positional Embedding)'을 사용했습니다. 하지만 최근(2027년 기준) Llama 등 최신 모델 실무에서는 이를 사용하지 않고 'RoPE(회전 위치 임베딩)'와 같은 상대적 위치 정보 기반 기술을 표준으로 사용합니다. 그 실무적/아키텍처적 가장 큰 이유는 무엇인가요?

### 💡 모범 답안 (정확한 이해를 위한 체크)

**답안 1**
트랜스포머의 핵심 엔진인 '어텐션(Self-attention) 메커니즘'은 문장 내의 모든 단어를 병렬로 한 번에 연산하기 때문에, 단어의 순서(어순)를 파악하는 기능이 아예 내장되어 있지 않은 '순서 맹(Position-agnostic)' 상태입니다. 따라서 위치 임베딩이라는 나침반을 따로 달아주지 않으면 모델은 "철수가 영희를 때렸다"와 "영희가 철수를 때렸다"를 전혀 구분하지 못하는 심각한 논리적 오류를 범하게 되기 때문입니다.

**답안 2**
256차원, 나아가 수천 차원이 넘는 고차원 벡터 공간(High-dimensional space)에서는 각 차원이 담고 있는 정보의 대역폭이 매우 큽니다. 수학적으로 이런 고차원 공간에서 두 벡터를 덧셈하더라도, 단어 본연의 의미적 특징(사과, 과일)과 공간적 위치 정보(3번째)가 서로 데이터를 덮어씌워 파괴하지 않고 독립적인 직교성(Orthogonality)을 유지하며 안전하게 결합된다는 것이 증명되었습니다. 덕분에 벡터를 이어 붙여 차원을 2배로 늘리는 것에 비해 GPU 메모리와 연산량을 절반으로 아낄 수 있는 막대한 효율성 이점을 얻게 됩니다.

**답안 3**
절대 위치 임베딩은 훈련할 때 설정해 둔 최대 문맥 길이(예: 4,096토큰)를 단 한 글자라도 넘어가게 되면, 4,097번째 자리에 대한 위치 텐서 값 자체가 아예 존재하지 않아 모델이 작동을 멈추거나 헛소리를 하게 되는 치명적인 '길이 연장 불가(Extrapolation 실패)' 단점이 있습니다. 반면 RoPE와 같은 상대적 위치 방식(각도 회전 등)을 사용하면, 수학적 규칙에 의해 위치가 결정되므로 훈련 시에 보지 못했던 수십만 단위의 엄청나게 긴 문맥(Long-context)이 주어져도 유연하게 거리를 계산하며 길이를 무한히 확장할 수 있기 때문입니다.
